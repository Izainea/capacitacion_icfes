Flujos de Trabajo Colaborativos

El Problema: Caos en la Rama Principal

Cuando varias personas trabajan directamente sobre la misma rama (ej. `main` o `master`), es fácil que se generen conflictos, se introduzcan errores sin revisión y el historial se vuelva un caos. Los flujos de trabajo establecen un conjunto de reglas para evitar estos problemas.

Estrategia de Ramificación: GitFlow

GitFlow es un modelo popular que define roles específicos para distintas ramas, organizando el ciclo de vida del desarrollo:

  • main: Contiene el código de producción. Solo se fusiona desde `develop` al preparar un nuevo lanzamiento.
  • develop: Es la rama de integración principal. Contiene los últimos cambios de desarrollo listos para el próximo lanzamiento.
  • feature/<nombre-feature>: Se crean a partir de `develop` para trabajar en una nueva funcionalidad (ej. `feature/nuevo-grafico-interactivo`). Una vez terminadas, se fusionan de vuelta a `develop`.
  • release/<version>: Se crea desde `develop` cuando se prepara un lanzamiento. Permite hacer ajustes finales sin detener el desarrollo de nuevas funcionalidades en `develop`.
  • hotfix/<issue>: Se crea desde `main` para corregir errores críticos en producción. Se fusiona tanto a `main` como a `develop`.

Pull Requests: La Clave de la Calidad

Un Pull Request (PR) o Merge Request (MR) es una solicitud formal para fusionar una rama en otra. Es el corazón del trabajo colaborativo en plataformas como GitHub, GitLab o Bitbucket.

El proceso es el siguiente:

  1. Un desarrollador termina su trabajo en una rama `feature`.
  2. Crea un PR para fusionar `feature` en `develop`.
  3. Otros miembros del equipo revisan el código, dejan comentarios y sugieren cambios.
  4. Se pueden ejecutar pruebas automatizadas para verificar que el nuevo código no rompe nada.
  5. Una vez aprobado, el PR se fusiona (merge), integrando los cambios de forma segura.

Manejando Conflictos de Fusión

Un conflicto ocurre cuando dos ramas han modificado la misma parte de un mismo archivo. Git no puede decidir qué cambio conservar y te pide que lo resuelvas manually.

Para resolverlo:

# 1. Asegúrate de tener los últimos cambios de la rama destino git checkout develop git pull origin develop # 2. Vuelve a tu rama y intenta fusionar los cambios git checkout feature/mi-feature git merge develop # 3. Git te avisará del conflicto. Abre el archivo en conflicto, # verás marcadores como <<<<<<<, =======, >>>>>>>. # Edita el archivo para dejar la versión final que deseas. # 4. Una vez resuelto, añade el archivo y completa el merge. git add <archivo-en-conflicto> git commit
